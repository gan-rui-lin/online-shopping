## 1 引言

### 1.1 项目背景

本项目为课程设计级别的全栈电商平台后端系统，面向 **买家（Buyer）**、**商家（Merchant）** 和 **管理员（Administrator）** 三类用户角色。系统实现了从商品上架、浏览搜索、购物车管理、下单支付、物流发货到售后评价的完整电商业务闭环，并在此基础上融入了 AI 智能特性（RAG 商品问答、AI 文案生成、智能评论摘要、Agent 代购、定时购物计划）。

### 1.2 项目目标

| 目标 | 说明 |
|------|------|
| 完整业务闭环 | 覆盖用户注册→商品浏览→加购→下单→支付→发货→收货→评价全链路 |
| 多角色协同 | 买家、商家、管理员三端协同，权限隔离 |
| 智能化体验 | 基于 AI 的商品问答、文案生成、个性化推荐、代购 Agent |
| 高数据一致性 | 库存乐观锁防超卖、订单超时自动取消、关键数据快照 |
| 可扩展架构 | 模块化设计，便于后续扩展微服务、分布式事务等 |

### 1.3 文档范围

本文档覆盖系统全部功能模块（共 15 个业务模块、27 张数据库表、60+ REST API 端点）的需求说明、需求建模和非功能性需求。

## 2 用户需求说明

### 2.1 系统角色定义

| 角色 | 标识 | 说明 |
|------|------|------|
| **买家（Buyer）** | `ROLE_BUYER` | 注册后的普通用户，可浏览商品、加购、下单、支付、评价、收藏、使用 AI 功能 |
| **商家（Merchant）** | `ROLE_MERCHANT` | 通过申请审核后获得的角色，可管理店铺、发布商品、处理订单、发货、回复评价 |
| **管理员（Administrator）** | `ROLE_ADMIN` | 平台运营方，可审核商家入驻、审核商品上架、管理分类、查看平台统计数据 |
| **系统（System）** | — | 定时任务自动执行，如订单超时取消、热门商品刷新 |

### 2.2 核心业务需求（User Story）

#### 2.2.1 认证与用户管理模块

| 编号 | User Story | 验收标准 |
|------|-----------|----------|
| US-AUTH-01 | 作为访客，我希望能注册账号，以便使用平台服务 | 提供用户名、密码、昵称、手机号注册；用户名和手机号全局唯一；密码 BCrypt 加密存储；注册后默认为 ROLE_BUYER |
| US-AUTH-02 | 作为用户，我希望能登录系统并获取凭证 | 用户名+密码验证通过后返回 JWT Token；Token 包含 userId、username、roles 信息；有效期 24 小时 |
| US-USER-01 | 作为已登录用户，我希望能查看和编辑个人资料 | 可查看昵称、头像、手机号、邮箱等信息；可修改昵称、手机号、邮箱、头像 |
| US-USER-02 | 作为已登录用户，我希望能修改密码 | 需提供旧密码验证；新密码经 BCrypt 加密后更新 |

### 4.4 性能需求

| 指标 | 要求 | 实现措施 |
|------|------|----------|
| API 响应时间 | 核心接口 < 500ms | Redis 缓存热点数据、数据库索引优化 |
| 并发能力 | 支持课设级别并发（~100 QPS） | HikariCP 连接池（20 连接）、Redis 连接池 |
| 库存一致性 | 不允许超卖 | 乐观锁 + version 字段、库存锁定机制 |
| 订单超时 | 30 分钟未支付自动取消 | Redis ZSet 延时队列 + 60s 定时扫描 |
| 分页查询 | 大数据量下保持稳定 | MyBatis-Plus 分页插件、覆盖索引 |

### 4.5 安全需求

| 需求 | 实现方式 |
|------|----------|
| **身份认证** | JWT 无状态认证，Token 有效期 24 小时 |
| **密码安全** | BCrypt 加密（强度因子 10），禁止明文存储 |
| **权限控制** | RBAC 模型 + Spring Security `@PreAuthorize` 方法级授权 |
| **CSRF 防护** | 因采用 JWT 无状态认证，CSRF 已禁用（RESTful API 场景安全） |
| **CORS 跨域** | 配置跨域策略，支持所有来源（开发环境），生产环境应收窄 |
| **SQL 注入** | MyBatis-Plus 参数绑定，无动态 SQL 拼接 |
| **数据隐私** | 支持匿名评价；手机号、邮箱字段索引但不公开暴露 |
| **会话管理** | 无状态 SessionCreationPolicy.STATELESS |
| **审计追踪** | 订单操作日志、库存变动日志、所有操作带时间戳和操作人 |

### 4.6 可靠性与可维护性

| 需求 | 实现方式 |
|------|----------|
| **逻辑删除** | 所有业务表采用 `deleted` 字段软删除，保留历史数据，支持数据恢复 |
| **数据快照** | 订单中的收货地址和商品信息为下单时快照，避免源数据修改影响历史订单 |
| **乐观锁** | SKU 库存更新使用 `version` 字段避免并发冲突 |
| **统一异常处理** | `GlobalExceptionHandler` 统一捕获业务异常、参数校验异常、认证异常 |
| **统一响应格式** | 所有 API 统一返回 `Result<T>` 格式 `{code, message, data}` |
| **模块化架构** | 按业务域拆分 modules，每个模块独立包含 Controller/Service/Mapper/DTO/VO/Entity |
| **时间戳追踪** | MyBatis-Plus MetaObjectHandler 自动填充 `createTime`、`updateTime`、`deleted` |
| **日志记录** | 开发环境启用 SQL 日志输出（StdOutImpl）；可按模块调整日志级别 |
| **API 文档** | Knife4j 自动生成 OpenAPI 3 文档，支持在线调试 |
